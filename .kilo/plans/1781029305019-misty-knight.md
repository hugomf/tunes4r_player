# YouTube seek-within-buffer assessment plan

## Format clarification

The YouTube streams being tested use **Matroska (MKV)**, not MP4/AAC. This changes the container-level seek considerations:

- Matroska uses a different structure than MP4 (no `moov` atom, uses EBML/Cues instead)
- Symphonia's MKV reader has different seek behavior than the MP4 reader
- The metadata is typically at the beginning of the file in Matroska (SeekHead + Info segments)
- Range requests for Matroska need enough initial data to contain the SeekHead/Cues for proper seeking
- The existing 512 KiB `HEADER_RESERVE` may be insufficient for Matroska metadata in longer files

## Key Matroska-specific code paths

- `rust/crates/core/src/audio/stream/handling.rs:169`
  - `probe_format()` uses `symphonia::default::get_probe()` which auto-detects format
  - For Matroska, the probe reads EBML header, SeekHead, and Segment Info

- `rust/crates/core/src/audio/decoder/seek.rs`
  - Need to verify how `seek_to_position` handles Matroska specifically
  - Matroska uses cluster-based seeking with Cues index

- `rust/crates/core/src/audio/stream/decorator/caching.rs`
  - `HEADER_RESERVE` is 512 KiB — may need to be larger for Matroska metadata

## Likely real-connection failure modes (Matroska-specific)

1. **Insufficient header/cache for Matroska metadata**
   - Matroska files can have metadata (SeekHead, Cues) scattered or at the end
   - The 512 KiB `HEADER_RESERVE` may not capture enough of the EBML structure
   - When `CachedReader` starts at offset 0 after a seek, it may not have the Cues needed for efficient seeking

2. **Forward seeks in Matroska require Cues data**
   - Matroska's seeking relies on the Cues element to map timecodes to cluster positions
   - If Cues are not in the cached header, forward seeks may fail or be slow

3. **`is_seekable()` dependency on byte_len**
   - `ProbeMediaSource` only reports seekable if `byte_len_opt.is_some()` (handling.rs:161)
   - For YouTube streams, `source_byte_len` is set from `pipeline.total_bytes()` (commands.rs:73)
   - If `total_bytes()` returns `None`, probing may take a non-seeking path

4. **CachedReader starting at offset 0 after seek**
   - Same as before: `CachingDecorator::open(Some(ms))` creates `CachedReader` at offset 0
   - For Matroska, this means the reader must seek through potentially large files to find the right cluster
